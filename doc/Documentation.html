code splits into two main parts client side ( react ) and server side (node js), client consist of public for files and src for source code, and source code folder itself consist of 4 folder, api for query strings and apollo setup, component for view components and utilities for custom logical functions.

server side consists of database folder for mongoose database setup, graphql for graphql custom resolvers and schema, and models for mongoose data models

UI components

Ui are mainly is consists of agent field Ui part, and control center agent Ui part, the first one is able to fill forms and the second one has the ability to see filled form as table.

Git structure and branching

adding a new feature or solving a bug we mostly created new branch in our personal git repository and when code was completed we merge it with master branch but hardly any of this branches pushed to github or lasted long.

Technologies and libraries used

  • React: as the main frame UI framework
  • Ant design: React UI library for using React components
  • Apollo React: for communication between client and graphql server
  • Google map react: for loading maps
  • Express: Node.js server framework
  • Express-jwt: for authorization
  • Auth0: user management and authorization
  • Mongoose: for mongodb database connection
  • Graphql: for server requests
  • Graphql compose mongoose: create a direct connection between server resolvers and database
  • Morgan, Winston: loggers

Database

the project uses mongodb mlab online database

since app supposed to have live instance and using mlab was in project description using mongo as database wasn't really a choice.

Sql pros for project:

  • three of use has a theoritical and partially practical experience with sql
  • project is on inner organization app, number of users are not high so no-sql does not have advantage in here
  • data reliability is important

Mongodb pros for project:

  • the data structure in mongoose and graphql is very similar and make it easier to manage data form server

Process view

How the processes are separated

the different parts of architecture view, logic, data tried to be as separate as possible. in the client side graphql query string separated as api and formatter fuction are separated too as a logical part.

in the server side though there is not really a boundary between graphql and database because of the use of graphql-compose-mongoose and the similarity between data structures it took extra effort to implement a data access layer and separate them properly.

for authorization, react client login through auth0 and use its obtained token in header of requests to server and server endpoint is only available for login users and access level managed in resolvers

Communication mechanism

client communicate to graphql-express server with apollo-react, ui communicate with apollo directly, graphql-express server communicate with database with custom resolvers or predefined graphql-compose-mongoose resolvers.

Choices and challenges

because of the fact that project requirements were clear and choice of technologies where restricted by either project description or our knowledge, challenge was mostly about implementing new technologies that we weren't familiar with them and communication between them, other than that choice in data model for a optimized data fetching was changing and data models are changed several time in project time scoop.
